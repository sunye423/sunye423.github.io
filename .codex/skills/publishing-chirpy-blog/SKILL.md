# Publishing Chirpy Blog

## Inputs and scope

Accept one complete Markdown file. Locate the Chirpy repository from the user's path or by finding a root containing `_config.yml`, `Gemfile`, `_posts/`, and `tools/run.sh`. Keep the source file unchanged. Record `git status --short` before publication so pre-existing dirt can be distinguished from the generated post.

## Git boundary

Inspect the current branch and status. Default the allowed branch to `main`. If the repository is on another branch, stop unless the user explicitly requested that branch; pass an explicitly requested branch to `--branch` without switching to it.

Never run `git switch`, `git checkout`, `git worktree`, `git stash`, `git add`, `git commit`, or `git push`. Never create a branch, worktree, or stash. Leave all commit and push decisions to the user.

## Infer metadata

Inspect categories and tags in existing `_posts/` Front Matter first. Infer missing values semantically from the complete article while preserving every explicit source Front Matter value and the Markdown body:

- Preserve an explicit non-empty title. Otherwise use the first H1. If neither exists, infer a concise title from the opening paragraph and report that inference. If the article is empty or too thin to support a title, stop without writing and request a title or more content.
- Choose 1–2 broad Chinese categories consistent with the repository taxonomy.
- Choose 3–6 relevant lowercase ASCII kebab-case tags, reusing established tags when appropriate.
- Write a factual 60–120-Chinese-character description without promotional language.
- Preserve an explicit non-empty strict ASCII kebab-case `slug` exactly. Otherwise derive a stable, descriptive ASCII kebab-case slug.
- Preserve an explicit valid Front Matter date. If no explicit date exists, use the current Asia/Shanghai local time at execution in exact `YYYY-MM-DD HH:MM:SS +0800` format; never future-date an inferred value. If an explicit date is invalid, stop and request correction rather than silently replacing it.

For an article about building this Chirpy blog on macOS, use `categories: [博客, 建站]` and `tags: [jekyll, chirpy, macos, ruby, github-pages]`.

## Prepare

Create a fresh private temporary request path outside the repository. Use `mktemp` with a fixed, non-article-derived prefix; confirm the resulting file mode is `0600`:

```bash
request_path="$(mktemp -t publishing-chirpy-blog)"
chmod 600 -- "$request_path"
```

Write the JSON through the file-editing mechanism to the exact path printed by `mktemp`. Put every article-derived value only in this JSON data object, with exactly these keys and types:

```json
{
  "repo": "/absolute/path/to/chirpy-repository",
  "input": "/absolute/path/to/complete-article.md",
  "title": "在 macOS 上搭建 Chirpy 博客",
  "categories": ["博客", "建站"],
  "tags": ["jekyll", "chirpy", "macos", "ruby", "github-pages"],
  "description": "对文章内容的客观摘要。",
  "slug": "build-chirpy-blog-on-macos",
  "date": "2026-07-22 10:00:00 +0800",
  "branch": "main"
}
```

Never place JSON request values in shell assignments or interpolate them through a shell. The `request_path` variable contains only the path generated by `mktemp`, never article data. Never use a heredoc, `eval`, or a generated command string. Invoke the helper with request consumption enabled:

```bash
python3 "/Users/sunye/.codex/skills/publishing-chirpy-blog/scripts/prepare_post.py" \
  --request "$request_path" --consume-request
```

The helper deletes only that exact request file after reading it, on both success and failure. It never deletes the parent or a sibling. If Python cannot start and therefore cannot consume the request, delete only that exact request file with the file-editing mechanism; never use a glob, recursive deletion, or parent deletion.

Parse the single JSON receipt. Treat its `destination`, `permalink`, `metadata`, and `created` fields as authoritative. Stop on any error; do not overwrite a conflicting destination. Confirm the input source is unchanged and the generated Markdown body matches it.

## Validate

From the repository root, perform these steps in order:

1. Run `bundle check`. If dependencies are missing, stop, report them, and request authorization before installing anything.
2. Run `bash tools/test.sh` only after `bundle check` succeeds.
3. Treat the receipt `permalink` as data. Each explicit permalink segment has already been restricted to ASCII letters, digits, and the URL-safe `._~-` set, starts with an ASCII letter or digit, and cannot be `.` or `..`. Never interpolate the receipt `permalink` into a shell command. Remove its leading slash with a non-shell path API and verify `_site/<receipt permalink without leading slash>/index.html` through that API.
4. Start `bash tools/run.sh` only after the prior checks succeed.

Report validation failures verbatim. Do not install or update dependencies without user authorization.

## Preview

Build the preview URL by appending the authoritative receipt `permalink` to `http://127.0.0.1:4000`. Report it after the preview server starts. If port 4000 is occupied, report the conflict and wait for approval of a specific free port. Do not kill, replace, or interrupt the process using port 4000. After receiving approval, confirm the selected port is free and, from the repository root, run:

```bash
approved_port="<user-approved-free-port>"
bundle exec jekyll serve --livereload --port "$approved_port"
```

After that server starts, report the approved host and port with the same receipt `permalink` appended.

## Stop condition

Stop at local preview. Report the generated file, effective metadata, validation results, preview URL, and the scope of pre-existing dirty-tree entries. Explicitly state that no branch/worktree/stash was created and no commit or push was performed; leave commit and push to the user.
